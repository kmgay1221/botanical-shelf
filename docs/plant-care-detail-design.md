## 2. 状態定義

### 2.1 株（plant）の表示状態
| 状態 | 判定 | 表示 |
|---|---|---|
| 通常 | 下記以外 | 通常カード |
| 水やり期日 | 次回水やり日 ≦ 今日 | 💧チップ（当日=「今日」、超過=「n日超過」を赤系にせず--waterで） |
| 断水中 | 当月のマスタ間隔が null | サムネdim＋🌙チップ＋「〜○月」 |
| アーカイブ | archived=true | 一覧から非表示。設定>アーカイブ済みから閲覧・復元・完全削除 |

### 2.2 「今日やる」対象の判定（毎回オンザフライ計算）
```
対象 = アーカイブでない ∧ 当月がnullでない ∧ 次回水やり日 ≦ 今日(JST)
次回水やり日 = 最新のwatering記録の日時(JST日付) + マスタ間隔[サイズ][記録月]
水やり記録が1件もない株 = 登録日 + マスタ間隔[サイズ][登録月]
```
- 期日超過は当日と同じ「今日やる」扱い（過去分を積み上げない）
- 肥料: fertilizer_months に当月が含まれ、最新fertilizer記録から fertilizer_interval_days 経過で対象

### 2.3 ストリーク
「その日の終了時点（23:59 JST）で、期日超過の株が1つもない日」の連続日数。cron実行時に日次判定して profiles.streak_count を更新（都度全履歴を再計算しない）。株が0の期間はカウント継続。

### 2.4 断水明け
当月null → 翌月がnullでない境界。翌月1日のユーザー通知時刻に「（株名）が目覚めました。様子を見て水やりを再開してください」。dormancy_wake として notification_logs で重複防止。

## 3. 主要フローの例外系

| # | ケース | 挙動 |
|---|---|---|
| E1 | 株が0（初回） | ホーム/棚に空状態:「まだ株がいません。＋から最初の一株を迎えましょう」＋追加への導線 |
| E2 | 今日の対象が0 | ホームに「今日の水やりはありません 🌿」＋次の予定（最短の株と日付）を表示 |
| E3 | 同日2回目の水やり | 確認ダイアログ「今日は記録済みです。もう一度記録しますか？」 |
| E4 | 「すべて完了」タップ | 確認「n株に水やりを記録します」→ 一括insert → 失敗分があれば株名を列挙して再試行案内 |
| E5 | 記録の編集/削除 | 本人のみ。実行後に次回日を再計算。最後のwatering記録を削除した場合は1つ前の記録（なければ登録日）基準 |
| E6 | 完全削除 | 「アーカイブ（推奨）/ 完全に削除」の2段階。完全削除は株名入力の二重確認 → care_logs・Storage写真を連鎖削除。復元不可の明記 |
| E7 | 写真アップロード失敗 | 記録自体は保存し「写真の保存に失敗しました。詳細画面から追加できます」 |
| E8 | 写真50枚超過 | 「この株の写真が上限（50枚）です。古い写真を削除してから追加してください」でブロック |
| E9 | AI生成失敗/タイムアウト | 「生成に失敗しました。もう一度お試しください」＋再試行。レート制限超過は「本日の生成回数上限（10回）に達しました」 |
| E10 | AI生成が既存種と同名 | unique制約でエラー → 既存レコードを提示し「この種は登録済みです。こちらを使いますか？」 |
| E11 | 生成JSONが不正 | zod検証NG → ユーザーには E9 と同文言、詳細はサーバーログ |
| E12 | 天気API失敗 | 注記なしで通常表示（機能に影響させない） |
| E13 | 通知購読が拒否/未対応 | 設定に「通知は許可されていません」表示＋iOSはホーム画面追加ガイドへ誘導。アプリ内の今日やるリストで代替 |
| E14 | オフライン | 参照はエラートースト「オフラインです」。オフライン書き込み同期は非対応（仕様） |
| E15 | 許可外メールでログイン試行 | 「このアプリは招待制です」（リンクは送らない） |
| E16 | 月替わり跨ぎ | 次回日計算は常に「水やりした日の月」の間隔を使用（要件3.4.1）。表示中に日付が変わった場合はフォアグラウンド復帰時に再取得 |

## 4. API設計（Next.js Route Handlers）

| Method/Path | 役割 | 認可 |
|---|---|---|
| POST /api/species/generate | AI種プロファイル生成（Gemini呼び出し、zod検証、レート制限10回/日） | ログイン必須 |
| POST /api/species/register | 生成結果の登録（unique衝突時は既存を返す） | ログイン必須 |
| GET  /api/weather | ユーザー地点の当日湿度・降水・最高気温（Open-Meteo、1時間キャッシュ） | ログイン必須 |
| POST /api/push/subscribe, /unsubscribe | Web Push購読管理 | ログイン必須 |
| GET  /api/cron/hourly-notify | 毎時Cron。該当時刻ユーザーへ日次通知＋断水明け通知＋ストリーク日次更新 | CRON_SECRET ヘッダ検証 |

- 上記以外のCRUD（plants, care_logs, profiles）はSupabaseクライアント直（RLSが認可を担保）
- service_role キーはcronとStorage削除処理のみで使用。クライアントに露出させない

## 5. 通知処理の詳細（/api/cron/hourly-notify）

```
毎時0分に実行（vercel.json: "0 * * * *"）
1. now(JST) の「時」を取得 → notify_hour が一致し notify_enabled のユーザーを抽出
2. ユーザーごとに:
   a. notification_logs に (user, 'daily', 今日) があればスキップ
   b. 今日やる対象を算出（§2.2）。0件なら送らない（ログも残さない）
   c. 天気取得 → 注記文を組み立て（失敗時は注記なし）
   d. 「今日の水やり: 白鯨一号 ほか2株」＋注記 をWeb Push送信 → ログ記録
   e. 断水明け該当株があれば dormancy_wake を同様に送信（設定OFFなら送らない）
3. 全ユーザーのストリーク日次更新（前日終了時点の判定）
```
- 無効になった購読（410 Gone）は push_subscriptions から削除

## 6. データ処理の細目

- **日付の正**: サーバー・クライアントとも Asia/Tokyo で日付文字列(YYYY-MM-DD)に正規化してから比較。UTCのままの比較を禁止
- **写真**: browser-image-compression で長辺1080px・初期品質0.8、300KB超なら品質を下げ再圧縮。パス: `plant-photos/{user_id}/{plant_id}/{uuid}.webp`
- **図鑑の収集判定**: `exists(plants where owner=me and species_id=X)`（アーカイブ含む）
- **成長ギャラリー**: その株の photo_url 付き care_logs＋植物本体写真から最古と最新
- **リズムバー正規化**: `height% = (その株サイズの年内最小間隔) / interval * 100`、null は固定低バー
- **地域デフォルト**: 東京（lat 35.681, lng 139.767）。設定で市区町村名検索（ジオコーディングはOpen-Meteo Geocoding APIを使用）

## 7. 非対応事項（仕様として明記）

オフライン編集同期 / 複数株の一括登録 / 腰水モード / 個体別間隔上書き / AI写真診断（以上Phase 2）/ 多言語 / タブレット最適化
